### Summary
The `ferrotick-strategies` crate provides a solid foundation for defining trading strategies, generating signals, and applying composite signal logic. The use of traits for the core `Strategy` and `PositionSizer` establishes a good modular structure. However, there are significant architectural flaws and performance issues that must be addressed before this crate can be considered production-ready. Specifically, the strategy implementations suffer from unbounded memory growth leading to O(N²) complexity, and the composite signal generation logic contains a critical indexing bug caused by missing provenance data in the `Signal` struct. Additionally, order generation is tightly coupled to signal generation, bypassing the dynamic position sizing framework.

### Critical Issues (must fix)
- [ ] `crates/ferrotick-strategies/src/strategies/rsi_reversion.rs:59` — The `self.closes` vector grows unbounded by pushing every bar's close. Since `compute_rsi` is called on the entire history every bar, this creates an O(N²) performance bottleneck and a memory leak. *Suggested fix:* Use a `VecDeque<f64>` and limit its size to a sensible maximum lookback (e.g., `period * 5`). When passing to `compute_rsi`, use `.make_contiguous()`. 
- [ ] `crates/ferrotick-strategies/src/strategies/macd_trend.rs:65` — Same unbounded `closes` vector issue leading to O(N²) complexity and memory leaks. *Suggested fix:* Apply bounded `VecDeque<f64>` logic similar to above.
- [ ] `crates/ferrotick-strategies/src/strategies/bb_squeeze.rs:63` — Same unbounded `closes` vector issue leading to O(N²) complexity and memory leaks. *Suggested fix:* Apply bounded `VecDeque<f64>` logic similar to above.
- [ ] `crates/ferrotick-strategies/src/signals/composite.rs:76` — The `combine_weighted` method enumerates over the output `signals` vector and uses the `idx` to fetch weights from `self.generator.strategy_names()`. However, `generator.on_bar()` uses `filter_map`, meaning it drops strategies that return `None` (e.g., during their warmup period). This causes the indices to mismatch, applying the wrong weights to the wrong signals. *Suggested fix:* Add a `source`/`strategy_name` field directly to the `Signal` struct so provenance is explicitly tracked, rather than relying on brittle vector indices.
- [ ] `crates/ferrotick-strategies/src/traits/strategy.rs:13` — The `Signal` struct lacks a `source` or `strategy_name` field. Downstream consumers cannot identify which strategy generated the signal. *Suggested fix:* Add `pub strategy_name: String` to the `Signal` struct and populate it inside the strategy's `on_bar` method using `self.name()`.

### Important Issues (should fix)
- [ ] `crates/ferrotick-strategies/src/traits/strategy.rs:41` — The `Strategy` trait couples signal generation (`on_bar`) with order generation (`on_signal`). Strategies currently hold a hardcoded `order_quantity` state, which they use to construct `Order`s. This entirely bypasses the `PositionSizer` trait implementations (`Kelly`, `Volatility`, etc.) located in `src/sizing/position.rs`. *Suggested fix:* Decouple `Strategy` to strictly emit `Signal`s. Remove `on_signal` from the trait. Move order generation to a dedicated `PortfolioManager` or `ExecutionEngine` that takes a `Signal` and a `PositionSizer` to build `Order`s dynamically based on portfolio equity.
- [ ] `crates/ferrotick-strategies/src/signals/composite.rs:73` — `self.generator.strategy_names()` creates and allocates a new `Vec<String>` on every single bar evaluation, creating unnecessary garbage. *Suggested fix:* Cache the strategy names during `CompositeSignalGenerator::new()` or iterate over references using a method like `generator.strategy_name(idx) -> &str`.
- [ ] `crates/ferrotick-strategies/src/dsl/validator.rs:49` — The `validate_strategy_spec` function checks if `value` is finite, but completely fails to validate the `operator` (should be >, <, ==, etc.), the `indicator`, and the `action`. *Suggested fix:* Add strict validation to ensure `rule.operator` matches a known set of operators and `rule.action` maps to valid signal actions.

### Minor Issues (nice to have)
- [ ] `crates/ferrotick-strategies/src/signals/composite.rs:52` — In `combine_majority`, if `signals` only contains `Hold` signals, `buy` and `sell` remain `0`, and the `action` becomes `Hold`. However, the strength calculates to `0.0`. This is functionally correct, but dividing by `signals.len()` where `signals` only contains `Hold` feels semantically odd for a "majority" logic.
- [ ] `crates/ferrotick-strategies/src/sizing/position.rs:105` — `KellySizer` sets a hard minimum on `win_loss_ratio` of `0.01` to prevent division by zero (`win_loss_ratio.max(0.01)`). It might be better to return an allocation of `0.0` early if the win/loss ratio is exactly zero or negative to avoid silent unexpected math distributions.

### Recommendations
- **Incremental Indicators:** Consider rewriting the indicator functions in `ferrotick-ml` to support incremental updates (pushing a single new `f64` and receiving the updated value) rather than computing over a slice. This would drastically improve the efficiency of the event-driven strategy evaluations and eliminate the need for buffering `VecDeque`s entirely.
- **Risk Management Integration:** The `dsl/parser.rs` defines a `RiskManagementSpec` with stop-loss and take-profit parameters, but these are currently completely unused by the underlying Rust strategies. You should add support for generating bracket orders or managing stop signals directly within the strategies or via an execution wrapper.

### Positive Findings
- The core math inside `sizing/position.rs` for `Kelly`, `Volatility`, and `Percent` is properly guarded against zero/negative values and correctly bounds output distributions.
- The use of `thiserror` for the `StrategyError` domain ensures clean error propagation.
- The YAML DSL syntax design is clean and maps nicely into the `serde` structs with appropriate `deny_unknown_fields` and sensible defaults.
- Good use of robust enum variants (`SignalAction`, `OrderSide`, `PositionSizingMethod`) to explicitly type the domain instead of relying on magic strings or booleans.
